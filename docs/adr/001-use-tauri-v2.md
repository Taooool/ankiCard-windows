# ADR-001: 选用 Tauri v2 作为桌面客户端跨平台开发框架

## 1. 背景 (Context)

在构建 AnkiCard 记忆卡片客户端时，我们需要一个跨平台的桌面端容器来运行应用。我们需要在后台运行定时轮询线程以控制窗口的主动弹出，同时需要能够做到内存占用极低且能在 Windows 操作系统上无缝静默运行。

---

## 2. 备选方案 (Alternatives)

我们评估了以下三种桌面开发技术方案：

1. **Electron**
   - *优势*：生态极其庞大，前端交互与底层 API 库丰富。
   - *劣势*：打包体积巨大（通常在 80MB - 120MB 以上），内存占用非常高（即使是空白窗口也需要占用至少 100MB 内存），这对于一个极简、需要常驻后台的定时弹窗提醒软件来说是不能接受的。
2. **Wails (Go + Webview)**
   - *优势*：基于 Go 语言，编译产物小，内存占用低。
   - *劣势*：对多窗口控制和动态多窗口的管理支持不如 Tauri 完善，且在 Windows 系统的 API 集成度与打包便利度上稍逊。
3. **Tauri v2 (Rust + Webview2)**
   - *优势*：
     - 打包体积极小（Release 状态通常仅为 3MB - 5MB 左右）。
     - 内存占用极低（常驻后台仅占用 30MB 左右）。
     - 基于 Rust 提供极强的安全保障与系统级异步运行时（Tokio），便于编写高性能的后台每秒状态轮询线程。
     - 支持细粒度的权限和窗口生命周期管控，与 Windows 原生 Webview2 结合完美。

---

## 3. 决策 (Decision)

我们最终决定采用 **Tauri v2** 作为 AnkiCard 的跨平台桌面客户端开发框架。

---

## 4. 影响与后果 (Consequences)

### 正面影响
- 极小化客户端的体积，使用 NSIS 和 MSI 安装打包速度极快。
- 极大地降低了用户在日常使用电脑时的后台内存负担。
- 可以利用 Rust 的 Tokio 异步能力和强类型检查机制，编写线程安全的本地状态管理器（`AppState`）。

### 负面影响（需要妥协与规避的问题）
- Tauri v2 引入了严格的 Command 权限机制，前端新增调用的接口（如批量导入、批量删除）必须配置到 `permissions/app.toml` 中，增加了开发时的权限配置心智成本。
- Windows WebView2 启动第一帧会有“闪白”现象，我们需要显式在 Rust 后端为新创窗口指定默认的 Webview 底色。
