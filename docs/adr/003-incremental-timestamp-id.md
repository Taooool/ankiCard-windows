# ADR-003: 使用毫秒递增时间戳作为批量导入卡片的唯一 ID

## 1. 背景 (Context)

AnkiCard 采用时间戳（毫秒级）的字符串表示作为卡片的唯一标识符 `id`，以便在卡片列表和复习记录中作为主键进行查找、编辑和删除操作。

在实现“一键批量导入卡片”功能时，我们需要在一瞬间（通常在几毫秒的同一个 CPU 循环内）把成百上千张卡片解析并推入数据库。如果所有的卡片都直接调用当前系统时间戳获取方法，会因为 CPU 执行速度过快而获得完全相同的毫秒时间戳，从而导致新导入的所有卡片主键 ID 完全重复冲突，引起后续的 CRUD 操作及主键索引产生严重混乱。

---

## 2. 备选方案 (Alternatives)

1. **引入 UUID v4 / 随机哈希生成库**
   - *优势*：每次生成的 ID 绝对全局唯一。
   - *劣势*：需要为 Rust 后端引入额外的第三方依赖库（如 `uuid` 库）。此外，无序乱序的 UUID 作为 ID 相比于有序的时间戳 ID，失去了隐式的“按创建顺序排序”的天然属性，同时也不符合目前已有的时间戳 ID 设计规范。
2. **在导入循环中执行线程睡眠阻塞**
   - *优势*：不需要修改 ID 生成逻辑，直接强制程序休眠 1ms 避开同一时间戳。
   - *劣势*：极度浪费性能，且在主线程或异步运行时中使用线程阻塞休眠会引起前端 IPC 调用的严重卡顿或假死，是极其糟糕的代码反模式。
3. **获取单次起始时间戳并在循环体内部自增递增（Millisecond Incrementing）**
   - *优势*：
     - 在执行导入命令 `import_cards` 的那一刻，仅获取一次系统当前时间戳 `now` 作为基准起点。
     - 在遍历要添加的卡片列表时，每推入一张卡片，将 `now` 变量进行 `now += 1` 毫秒级的手动递增。
     - 导入结束后，卡片在保留有序性的同时，保证了彼此的毫秒级 ID 绝对唯一。
     - 无任何外部依赖，执行速度在纳秒级别，性能极佳。

---

## 3. 决策 (Decision)

我们最终决定采用 **获取单次起始时间戳并在循环体内部进行自增（Millisecond Incrementing）** 的轻量级算法设计方案。

---

## 4. 影响与后果 (Consequences)

### 正面影响
- 彻底规避了批量导入文件时卡片主键 ID 的碰撞问题。
- 卡片在主列表展示时，由于主键 ID 的毫秒级微小差值依然保留了创建时的相对先后顺序，可方便以 ID 作为排序退化退路。
- 零依赖、零等待、零性能开销。

### 负面影响（需要妥协与规避的问题）
- 批量导入的卡片其物理创建时间戳虽然比实际物理时间稍快几毫秒到几十毫秒，但这完全不影响任何业务逻辑以及用户的视觉体验。
