# ADR-002: 选用本地 JSON 扁平文件作为数据持久化层

## 1. 背景 (Context)

AnkiCard 作为一个极简的本地记忆卡片软件，必须保证用户创建的卡片信息（Front、Back、熟练度、弹出次数等）以及应用的配置参数（如提醒间隔、是否启用）在应用重启、断电或重装时得以持久化保存。

我们需要决定选用何种本地存储介质。

---

## 2. 备选方案 (Alternatives)

1. **关系型数据库 SQLite**
   - *优势*：支持复杂的 SQL 查询、事务控制，能在大批量数据下保持优秀的查询性能。
   - *劣势*：对于仅有几百到几千张卡片规模的极简应用来说过于臃肿，需要引入额外的 Rust 数据库连接库（如 `rusqlite` 或 `sqlx`），这会显著增加 Rust 编译体积以及引入数据库文件迁移和锁冲突的管理成本。
2. **Key-Value 存储 (如 RocksDB 或 sled)**
   - *优势*：极速的单键读写能力。
   - *劣势*：通常需要编译复杂的 C/C++ 依赖，并且数据不是以纯文本明文存储，不便于用户手动查看、修改或跨平台手动备份。
3. **本地 AppData 目录下的扁平 JSON 文件 (`cards.json` 与 `config.json`)**
   - *优势*：
     - 数据量小（千张卡片仅占几十KB字节），直接在应用启动时一次性加载进内存即可，查询性能极佳。
     - 数据以明文 JSON 保存，可读性极高。用户或 AI 助理可以非常方便地手动编辑、导入或通过网盘云备份。
     - Rust 内置的 `serde_json` 库解析性能极高，没有引入外部底层复杂数据库的编译开销。

---

## 3. 决策 (Decision)

由于卡片库的应用场景通常针对个人学习，数据体量不大（几千张卡片已属极限），我们最终决定采用**本地 AppData 目录下的扁平 JSON 文件**作为应用的数据存储层。

具体拆分为两个文件：
- `cards.json`：存放卡片实体数组。
- `config.json`：存放应用定时设置和开关参数。

---

## 4. 影响与后果 (Consequences)

### 正面影响
- 数据读取与持久化逻辑极其直白，开发调试方便，出错时只需打开本地 JSON 文件即可秒定位。
- 零外部运行时依赖，打包编译后体积极小。
- 用户可以非常直接地分享或拷贝 `cards.json` 实现本地数据迁移。

### 负面影响（需要妥协与规避的问题）
- 数据是全量读取进内存，也是全量 Pretty 写入覆盖文件的。当卡片数据达到数十万级别时可能会产生写入延迟（不过当前应用场景不需要考虑这一极端场景）。
- 并发读写冲突风险：我们通过在后端将 `AppState` 内存缓存用 `Mutex`（互斥锁）包裹，并在有写行为的 Command 中独占锁定状态后执行写盘，从而在多窗口并发调用的情况下规避了并发写冲突风险。
